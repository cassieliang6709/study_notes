# 从最小 RAG 到复杂 RAG：每一层多了什么

当我把基础 RAG 跑通之后，很快发现它在真实场景下的问题：有些问题检索结果很准，有些完全跑偏；有些用户的提问方式和文档里的表达完全不匹配；有些问题需要查好几个地方才能回答。

这就是为什么"复杂 RAG"存在的原因。它不是为了复杂而复杂，而是真实场景里把每一层的问题暴露出来之后，一层一层加上去的解决方案。

## 最小 RAG 的样子

先把最小版本说清楚：

```
文档 → chunk → embedding → 向量库
                                ↓
用户提问 → query embedding → 检索 top-k chunk → 拼进 prompt → 生成答案
```

这已经能工作了。但它有几个明显的薄弱点。

## 第一层问题：数据本身

最容易被忽略的问题是数据质量。

原始文档通常不干净：PDF 解析有乱码、表格变成了一行字符串、标题和正文混在一起。如果这些噪声进了向量库，检索出来的 chunk 质量就很低。

所以复杂 RAG 第一步是做 **数据清洗**，在切块之前把文档规范化。这不性感，但往往是效果差距最大的地方。

第二个问题是 **切块策略**。

最简单的方法是按固定字符数切，但这会切断句子，导致每个 chunk 缺少上下文。更好的方式是按语义单元切（段落、章节），同时加一些重叠，让相邻 chunk 之间有连续性。

chunk 太大的问题是：检索粒度粗，一个 chunk 里包含的话题太多，语义向量会被稀释，检到的内容噪声大。

chunk 太小的问题是：每个 chunk 缺少上下文，单独拿出来没有意义，需要大量拼接才能回答。

找到合适的 chunk 大小是需要实验的，不存在通用最优解。

## 第二层问题：检索本身

用户的提问方式，往往和文档里的表达不一样。

用户可能问"这个模型训练要多久"，但文档里写的是"训练耗时约 48 小时"。关键词不重叠，纯向量检索可能找不到。

**Query rewriting** 就是在检索之前，先把用户的问题改写成更适合检索的形式。常见做法是用 LLM 生成多个改写版本，然后对每个版本都检索，再合并结果。

另一个问题是：向量相似度只看语义接近，不一定看内容相关。有时候语义相近但内容无关的 chunk 会排在前面，真正有用的 chunk 反而被埋了。

**Reranking** 是在初步检索之后，用一个更精准的模型对候选 chunk 重新打分排序，把真正相关的拉上来。这一步计算代价更高，所以只在 top-k 的小范围内做。

纯向量检索还有一个问题：对精确关键词匹配不敏感。如果用户问的是一个专有名词或者产品名，语义向量可能找不到精确匹配。

**混合检索**（hybrid search）就是把向量检索和关键词检索（比如 BM25）同时跑，然后合并排名。我在做 MemoryOS 的时候就用了这个方案：纯向量搜索在概念检索上好，关键词过滤在精确时间和标签约束上好，两个并行跑，按权重合并分数。

## 第三层问题：复杂问题

有些问题不是一次检索能回答的。

比如"比较 A 方法和 B 方法在低资源场景下的表现"——这需要找到关于 A 的内容、关于 B 的内容、关于低资源场景的定义，然后把三者综合。

**Planner / Executor 结构**是为这类问题设计的。

Planner 先把问题拆成子问题：要回答这个问题，我需要知道哪几件事？然后 Executor 对每个子问题分别检索，收集所有结果，最后再综合生成答案。

这更像一个多步搜索过程，而不是一次性的"检索 + 回答"。

## 第四层问题：评估

基础 RAG 很难评估：你看着输出觉得还行，但不知道有多少次检索是真的找到了对的内容，有多少次答案看起来合理但其实用的是错误材料。

复杂 RAG 会加入 **evaluation 层**，常见的评估指标有：

- **检索准确率**：找到的 chunk 和问题真的相关吗
- **答案忠实度**：回答是否真的基于检索内容，还是模型在编
- **答案相关性**：回答有没有真正回答问题

有了评估，才能系统地改进，而不是靠直觉猜哪里出了问题。

## 怎么看待这些层

我现在看复杂 RAG，会把它理解成一组"当简单方案不够用时的应对措施"：

| 问题 | 对应方案 |
|---|---|
| 数据质量差 | 清洗和规范化 |
| chunk 切得不好 | 语义切块 + 重叠 |
| 用户提问和文档不匹配 | Query rewriting |
| 检索结果有噪声 | Reranking |
| 精确关键词匹配失败 | 混合检索 |
| 复杂多步问题 | Planner / Executor |
| 不知道哪里出了问题 | Evaluation |

每一层都不是默认就要上的，而是先发现问题，再针对性地加。

## 小结

从最小 RAG 到复杂 RAG，不是功能堆砌，而是一个问题列表一条一条被发现、被解决的过程。

理解这个演进过程比死记架构图更有用。下次你的 RAG 系统效果不好，先问自己：是哪一层出了问题？再决定加什么。
