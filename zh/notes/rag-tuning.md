# RAG 调优的正确顺序：不要上来就换模型

做 MemoryOS 的时候，有一段时间检索效果一直不好。我的第一反应是换一个更好的 embedding 模型，换了之后效果依然很一般。后来我认真去看了检索结果，发现问题根本不在 embedding，而是 chunk 切得太碎，每个 chunk 被切到只有半句话，语义完整性很低。

把切块策略改了之后，什么都没换，效果立刻好了很多。

这件事让我意识到：RAG 调优有一个正确的顺序，不遵循这个顺序，很容易在错误的地方浪费时间。

## 先诊断，再动手

在改任何东西之前，先看检索结果。

把你测试的问题对应的检索 chunk 打出来，逐条看：

- 检索到的内容和问题相关吗？
- 相关的内容排在前面吗？
- chunk 本身语义完整吗，还是被切断了？
- 原始文档有没有解析错误或噪声？

如果检索结果是对的，但答案还是不好，那才去看生成层。如果检索结果本身就有问题，那不管怎么改 prompt 或换模型都没用。

**大部分情况下，RAG 系统效果差，根源在检索而不是生成。**

## 调优顺序

### 第一步：看数据和 chunk

**先看数据质量。**

文档是怎么解析进来的？如果是 PDF，有没有解析乱码？表格有没有被破坏？如果是网页，有没有抓到一堆导航栏和 footer 的噪声？

这些问题很枯燥，但影响非常大。我见过有人花了很多时间调整检索策略，最后发现是因为 PDF 解析出来的文本每隔几行就有一个随机的页码数字混在正文里。

**然后看 chunk 策略。**

把几个 chunk 单独打出来看。它们作为独立的段落有意义吗？如果看起来像截断的句子，chunk 切得太小了。如果每个 chunk 都包含好几个不同的话题，chunk 切得太大了。

调 chunk 大小和重叠率，比换 embedding 模型便宜很多，而且往往更有效。

### 第二步：看检索策略

**top-k 设多少？**

top-k 太小：可能关键信息没进上下文窗口。top-k 太大：噪声太多，模型反而被干扰。

通常先从 3-5 开始，看效果再调。

**是否需要混合检索？**

如果你的问题里有很多专有名词、产品名、人名，纯向量检索可能会漏掉精确匹配。这时候加关键词检索（BM25）和向量检索并行跑，效果会好很多。

**是否需要 reranking？**

如果检索结果的数量已经够了，但排序不对（最相关的没排在最前面），可以加 reranking。这一步代价比较高，不要默认加，先判断是否真的是排序问题。

### 第三步：看 prompt

到了这一步，才去调生成层。

**上下文放在哪？**

很多研究发现，LLM 对 prompt 里"开头"和"结尾"位置的内容更敏感，中间的内容更容易被忽略。如果你的 top-k 检索结果里最相关的那条在中间，可以试着把它调到最前或最后。

**有没有让模型引用来源？**

在 prompt 里明确要求模型注明答案来自哪个 chunk，有两个好处：一是方便你验证答案是否真的有依据，二是模型被要求引用时，编造的倾向会降低。

**当检索内容不够时，让模型说"我不知道"。**

这个要明确写在 prompt 里。模型默认倾向于给出一个答案，即使它的依据不充分。

### 第四步：才是换 embedding 模型

换 embedding 模型是代价最高、收益最不确定的操作之一，因为你需要重新算所有 chunk 的向量，而且效果改善不一定显著。

在数据、chunking、检索策略、prompt 都调过之后，如果还有问题，再考虑换 embedding 模型。

通常要换的信号是：语义检索的质量在概念理解层面明显很差，而不只是排序问题。

## 一个简单的诊断框架

当你的 RAG 系统效果不好时，问自己这几个问题：

1. 检索到的 chunk 和问题相关吗？→ 不相关就先看数据和 chunking
2. 相关 chunk 排在前面了吗？→ 排序不对就看 reranking
3. Chunk 的语义完整吗？→ 不完整就调 chunk 大小和重叠
4. 用户的问题表达方式和文档差异大吗？→ 考虑 query rewriting
5. 上下文质量没问题，但答案还是差？→ 才去看 prompt 和生成

## 小结

调 RAG 系统的直觉是：**哪里是瓶颈，就在哪里下手，不要随机尝试。**

最常见的错误是检索出了问题，却去优化生成；或者数据质量很差，却去换更好的 embedding 模型。

建立一个简单的评估集——哪怕只有二三十个问题——是让你能系统诊断问题、而不是靠感觉猜的最低成本方式。
