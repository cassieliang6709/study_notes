# RAG 是什么，以及为什么直接问大模型不够用

我第一次接触 RAG 的时候，以为它是个花里胡哨的工程组件，直到我真的开始做 MemoryOS，才理解它解决的是一个很基础的问题。

## 大模型的知识是固定的

大语言模型的知识来自预训练。训练结束，它知道的东西就固定下来了。你问它 2021 年前的事情，它可能答得不错；你问它昨天发生的新闻，它不知道；你问它你公司内部的文档，它更不知道。

这不是模型变笨了，而是它的知识本来就是参数里固化的记忆，没有动态获取新信息的能力。

另一个问题是幻觉。模型在找不到确定答案的时候，会倾向于"编"一个听起来合理的答案，因为它的训练目标是生成合理的下一个 token，而不是"不知道就说不知道"。这在需要事实准确的场景里是致命的。

## RAG 是什么

RAG 的全称是 Retrieval-Augmented Generation，检索增强生成。它的核心思路很简单：

**回答之前，先去找相关材料。再把找到的材料一起给模型，让模型基于材料来回答。**

最小版本的流程大概是这样：

1. 把你的文档切成小块（chunk）
2. 给每个 chunk 算一个向量表示（embedding）
3. 用户提问
4. 把问题也算成向量，去向量库里找最接近的 chunk
5. 把检索到的 chunk 拼进 prompt
6. 模型基于这些上下文生成回答

这样一来，模型回答的依据不再只是参数里的记忆，而是你明确提供的材料。这叫做 **grounded generation**，回答是有根据的，而不是凭空生成的。

## 和直接问大模型的区别在哪

直接问大模型：模型用参数里存的知识回答，知识截止日期固定，无法访问私有信息，幻觉风险高。

加上 RAG：模型基于你提供的当前材料回答，可以接入私有文档、最新数据，回答更有据可查，也更容易溯源。

一个实际的例子：假设你有一堆研究论文，想问某个方法的细节。直接问大模型，它可能给你一个听起来对但实际上是编的答案。用 RAG，它会先在你的论文库里找到相关段落，然后基于那几段原文回答——你甚至可以让它注明来自哪篇论文的哪一段。

## RAG 适合什么场景

RAG 不是所有问题的解法。它特别适合：

- **知识更新频繁的场景**：新闻、报告、产品文档
- **私有知识库问答**：公司内部文档、个人笔记
- **需要引用来源的场景**：研究辅助、法律、医疗
- **避免幻觉要求高的场景**：任何需要事实准确的地方

它不太适合的场景：

- 纯创意生成（不需要外部知识）
- 需要复杂推理但知识库太小的情况

## 为什么检索质量比生成质量更重要

这是我学 RAG 最早搞清楚的一件事：检索和生成是两件事，而且检索更基础。

如果检索阶段找到的 chunk 根本不相关，或者包含太多噪声，那不管用多强的模型来生成，答案都会受影响。模型能做的最多是"在你给的材料里组织语言"，如果材料本身就错了，它没有办法凭空纠正。

相反，如果检索质量很好，就算用一个中等水平的模型，答案往往也还可以。

所以调优 RAG 系统时，第一步永远是看检索结果，而不是换更大的模型。

## 小结

RAG 解决的不是模型能力不够的问题，而是信息获取范围的问题。它让模型能够在回答时依赖你提供的外部知识，而不是只靠固化在参数里的记忆。

理解这一点之后，RAG 系统的很多设计选择就变得直觉了：为什么要切块、为什么要做向量检索、为什么要做 source attribution——都是为了让"检索到的材料"这一层尽量准确、可信、可追溯。
