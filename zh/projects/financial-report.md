# FinancialReport

包含数据接入、检索、校验和报告组装的财务数据平台。

## 项目概述

FinancialReport 是一个数据流水线和报告后端，负责接入异构财务数据（CSV、Excel、JSON），将其校验和规范化为统一 schema，并组装结构化报告供下游分析使用。核心挑战在于原始数据从来都不干净：不同客户的科目名称不同，编码不一致，不同数据源的列 schema 也各不相同。

## 技术栈

- **语言**：Python 3.11
- **数据层**：pandas、SQLAlchemy、PostgreSQL
- **校验**：Pydantic schema、自定义规则引擎
- **接入**：多格式解析器（CSV、Excel、JSON）
- **报告**：Jinja2 模板、PDF 导出

## 关键设计决策

**三层接入流水线** — 数据依次流经规范化 → 映射 → 校验，才能进入报告层。每一层有清晰的接口，失败是隔离的，可以在不重新处理上游步骤的情况下重试。

**基于规则的科目映射与异常路由** — hash map 处理精确匹配（O(1)，覆盖约 75% 的科目）；正则 fuzzy matching 处理命名变体。置信度低于阈值的条目路由到 exception log，而不是静默通过——系统不会猜测。

**边界处的 Pydantic 校验** — 所有接入记录在写入数据库前都通过严格 schema 校验。无效记录会被拒绝并带有结构化错误信息，而不是静默强制转换。

**报告组装与接入解耦** — 报告层只从经过校验和规范化的数据存储中读取，从不接触原始接入产物。这让数据模型保持稳定，报告模板也易于独立测试。

## 挑战

最棘手的是映射引擎的置信度阈值调优。阈值设太高，太多记录进入 exception log（复核慢）；设太低，坏映射会静默通过。最终解决方案是在标注好的 holdout 集上做校准，通过最大化 exception routing 决策的 F1 分数来调参，而不是随手选一个整数。

第二个挑战是编码问题。老旧会计系统的财务数据常常以 Latin-1 或 CP1252 编码，而非 UTF-8。规范化层现在通过 `chardet` 解析前检测编码，并将检测到的编码记录为元数据。

## 结果

- 映射准确率：调参后约 94% 的科目可被正确自动映射
- 数据准备时间：每个 engagement 从约 2 天（手工）降至约 3 小时
- Exception log 复核量：从 100% 的记录降至约 10% 需要人工判断
