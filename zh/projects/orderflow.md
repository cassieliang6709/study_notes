# OrderFlow

基于 Java Spring Boot 的订单全生命周期管理后端，涵盖库存和支付工作流。

## 项目概述

OrderFlow 是一个管理电商订单完整生命周期的后端服务：创建订单、预留库存、处理支付、管理履约状态转换，以及带回滚机制的取消操作。系统设计核心是正确性和一致性——每一次状态转换都是显式的、有日志的、可恢复的。

## 技术栈

- **语言**：Java 17
- **框架**：Spring Boot 3、Spring Data JPA
- **数据库**：PostgreSQL（事务、行级锁）
- **消息**：异步事件发布（供下游消费者使用）
- **测试**：JUnit 5、Mockito、Testcontainers

## 关键设计决策

**订单生命周期状态机** — 订单在固定状态集合中流转（`CREATED → RESERVED → PAID → FULFILLED → CANCELLED`）。状态转换在 service 层校验，无效操作（如对已取消订单付款）在进入数据库之前就被拒绝。

**乐观锁库存预留** — 库存预留采用两阶段设计：支付确认前为 `RESERVED`，确认后为 `COMMITTED`。这样在高并发下不需要持有长锁也能防止超卖。

**幂等支付处理** — 支付事件通过幂等键去重，客户端重试（网络故障或客户端重试）不会导致重复扣款。

**补偿事务回滚** — 取消操作触发补偿写入（释放库存、撤销支付预授权），而非硬删除，保留审计历史。

## 挑战

最难的问题是处理"支付 → 库存 → 履约"序列中的部分失败。如果支付成功但库存提交失败，系统需要在不产生用户可见不一致的情况下回滚支付。解决方案是 local saga pattern：每一步在执行前写入一条回滚记录，这样崩溃后后台 reconciler 可以完成或撤销未完成的事务。

## 结果

- 订单创建和状态转换：压测下 p99 延迟 <5ms
- 并发压力测试（100 个并发请求抢占数量为 1 的单个 SKU）：零超卖
- 所有失败路径均通过 Testcontainers + 模拟部分失败进行测试覆盖
