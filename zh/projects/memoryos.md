# MemoryOS

基于 FastAPI、React、PostgreSQL 和 pgvector 的检索型产品系统。

## 项目概述

MemoryOS 是一个个人知识管理系统：用户存储笔记、经历和知识片段，并通过语义搜索进行检索。后端负责接入、嵌入、向量存储和排序检索；前端提供简单的录入和搜索界面。核心设计目标是检索质量——不只是返回结果，而是返回正确的结果，并具备可解释的相关性。

## 技术栈

- **后端**：Python、FastAPI、SQLAlchemy
- **数据库**：PostgreSQL + pgvector 扩展
- **嵌入**：OpenAI `text-embedding-3-small`（可替换接口）
- **前端**：React、TypeScript
- **搜索**：混合检索 — 向量相似度 + 关键词过滤
- **部署**：Docker Compose

## 关键设计决策

**用 pgvector 而非独立向量数据库** — 将嵌入向量与关系型元数据一起存放在 PostgreSQL 中，避免了维护独立向量数据库的运维复杂度。`ivfflat` 索引在当前数据规模下足够。如果数据集增长到数百万条记录，迁移到独立向量存储也很简单，因为嵌入接口已经抽象化。

**混合检索** — 纯向量搜索在语义相近的结果上表现好，但会漏掉精确关键词查询。纯关键词搜索会漏掉同义改写和概念层次的匹配。系统并行运行两者，按加权分数合并结果，关键词过滤器在时间或标签约束上作为硬过滤条件。

**分块策略** — 长笔记按段落分块，重叠率 10%。每个 chunk 存储对原始文档的反向引用。检索返回 chunk，但 UI 始终展示完整文档上下文，用户不会只看到脱离来源的片段。

**嵌入模型作为依赖注入** — 嵌入客户端在启动时注入。从 OpenAI 切换到本地模型（如 `sentence-transformers`）只需改一个配置值，不需要重构调用处。

## 挑战

最难的检索问题是时间相关性：3 年前语义上非常接近的结果，往往不如上周中等相关的结果有用。在排序分数中加入时间衰减因子有所帮助，但合适的衰减系数需要根据真实用户反馈调参，而非凭公式推导。

第二个问题是分块粒度。块太小会丢失上下文；块太大会稀释嵌入信号。段落级分块加重叠是合理的默认值，系统将 chunk size 作为可配置参数暴露出来。

## 结果

- 语义搜索延迟：约 50k chunks 规模下 p95 <100ms（pgvector ivfflat，单节点 Postgres）
- 检索精度：在 200 条测试查询上，混合检索比纯向量检索在定性评估上有所提升
- 完整服务通过 `docker compose up` 一键启动
